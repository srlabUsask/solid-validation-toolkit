Method-level scoring rubric

You are scoring a single Java method extracted from a larger system. You are not inferring intent or domain meaning. You are scoring observable structural signals only.

Identifiers may have been consistently renamed (e.g., x1, x2). Do not infer semantics from names.

0 = Violated (clear structural evidence)
1 = Partial / Uncertain (mixed signals OR insufficient context)
2 = Compliant (clear structural evidence)
N/A = Use only if the method is non-Java code, garbled, or otherwise not assessable. Insufficient context is not N/A — it is score 1.

Be conservative. If the method alone is insufficient to make a confident call, score 1.

1. SRP — Single Responsibility (method cohesion)

Score whether the method does one cohesive responsibility.

Score 0 — Violated
  • Multiple distinct responsibilities present (e.g., validates + persists + logs), OR
  • Unrelated side effects mixed (e.g., updates state + UI + IO).
Score 2 — Compliant
  • One focused responsibility, OR
  • Delegates via at most 3 calls and contains no independent logic beyond delegation sequencing (thin orchestrator).
Score 1 — Partial / Uncertain
  • SRP judgement depends on class-level role or broader workflow not visible in this method alone.

2. OCP — Modification-risk signals (method-level proxy)

You are not declaring true OCP compliance for the system. You are scoring whether this method structurally encodes variation such that future requirements likely require modifying this method.

Score 0 — Violated
  • if/else or switch chain with 3+ branches encoding behavioral variants, OR
  • Hard-coded policy thresholds / magic constants that appear to represent changeable policy, OR
  • Repeated copy-modify style branches.
Score 2 — Compliant
  • Behavior is delegated to collaborators / polymorphic calls, and branching is minimal (0–2 branches) and not encoding variants.
Score 1 — Partial / Uncertain
  • You cannot see whether delegation is polymorphic / strategy-based, OR
  • Branching exists but could be simple control flow without representing an extensibility point.

3. DIP — Dependency Inversion (replaceable-dependency signals)

Score whether the method introduces tight coupling to replaceable concrete collaborators.

Score 0 — Violated
  • Instantiates a likely replaceable collaborator/service (e.g., a repository, network client, UI subsystem, persistence service) directly inside the method, OR
  • Uses static singletons / globals for replaceable collaborators, OR
  • Strong framework coupling that acts as a direct dependency (not just DTOs / events).
The following are NOT DIP violations:
  • Instantiating collections (ArrayList, HashMap, etc.), strings, primitives, simple data/value objects, iterators/enumerators
  • Creating Exceptions
  • Creating small helper objects local to the method, when not a replaceable external collaborator
Score 2 — Compliant
  • Dependencies are injected via parameters/fields (as abstractions), OR
  • No replaceable dependencies appear in the method.
Score 1 — Partial / Uncertain
  • A new X(...) appears, but you cannot determine whether X is a replaceable collaborator vs. a value/helper type.

Scoring discipline

  • Score the method as written, not the imagined system around it. If a judgement requires seeing the class, the inheritance hierarchy, or other modules, that's score 1, not score 0 or 2.
  • Do not infer semantics from method or variable names. Identifiers may have been renamed.
  • Test code is fair game. A test method that instantiates a service-like collaborator is still a DIP-0 signal at the method level.
  • Empty bodies should usually score 1 with the comment "empty body". They give no signal either way.
  • Use the comment field when a case is unusual, when you suspect the rubric does not quite cover it, or when you went back and forth on a decision.
