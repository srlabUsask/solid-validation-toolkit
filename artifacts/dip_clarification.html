DIP Rubric Clarification — Round 3 Annotation

Refinement of the Dependency Inversion Principle scoring criteria, based on PI review of round-1 and round-2 annotations. This addendum does not change the rubric; it clarifies how the existing criteria apply to common method-level patterns in Java.

1. Rubric — DIP scoring criteria (unchanged)

DIP (Dependency Inversion Principle), method-level scoring:
  • 0 — Violated: The method instantiates a replaceable collaborator or service directly, uses a static singleton for replaceable behaviour, or contains strong framework coupling that hard-codes a low-level implementation choice.
  • 1 — Partial / Uncertain: The method instantiates a type with new or otherwise creates a dependency, but it is unclear from the method alone whether the type is a replaceable collaborator (DIP=0 territory) or a value/helper/utility type (DIP=2 territory).
  • 2 — Compliant: The method has no replaceable dependencies (its dependencies arrive as parameters or fields, or it instantiates only value types, exceptions, JDK utilities, or collections), or its dependencies are injected from outside.

2. What DIP is actually asking

The Dependency Inversion Principle is about whether a method hard-codes the choice of a replaceable collaborator. The key word is replaceable. A collaborator is replaceable if there is a plausible alternative implementation that a different application would want to plug in.

A real-world example: a method that needs to send an email could use SmtpMailSender, SendgridMailSender, or a fake one for testing. The mailer is replaceable. If the method writes new SmtpMailSender() directly, that is a DIP violation — it has hard-coded a low-level choice.

By contrast, a method that needs to count items in a list using new ArrayList<>() is not making a "replaceable collaborator" choice. ArrayList is a value-like container; there is no "alternative ArrayList implementation" the method would plausibly swap to. This is not a DIP violation.

3. What is NOT a DIP violation

The following patterns are not DIP violations at method level, regardless of how many new keywords appear:

(a) Collections, maps, sets, and arrays.
new ArrayList<>(), new HashMap<>(), new HashSet<>(), new TreeMap<>(), new String[size]. These are containers, not replaceable services.
(b) Value types, DTOs, records, and primitives.
new String("..."), new Integer(...), new BigDecimal(...), new Date(), and any data-holding object that the method creates rather than collaborates with. A user-defined record or POJO is in this category if it carries data rather than behaviour.
(c) Exceptions.
throw new IllegalArgumentException(...), throw new IOException(...), new RuntimeException(message, cause). Exception types are not replaceable collaborators; they are part of the method's contract.
(d) JDK static utilities.
Files.copy(...), Files.readAllBytes(...), Arrays.asList(...), Objects.requireNonNull(...), String.format(...), Math.max(...), Collections.unmodifiableList(...). These are namespaced functions, not services with alternative implementations.
(e) Local helpers and iterators.
Small private helper objects that the method constructs and discards within its own scope. If the helper is not visible outside the method, it is not a replaceable collaborator.
(f) Constructor / factory bodies.
A method whose entire job is to construct an object often instantiates several other objects as part of that construction. This is the method's job, not a DIP violation. Builders, factories, and constructors get a pass here.

4. What IS a DIP violation

The following patterns are DIP violations at method level:

(a) Direct instantiation of a service or collaborator.
new EmailService(), new DatabaseConnection(url), new HttpClient(), new PaymentGateway(). Each of these has plausible alternative implementations the method could plug in via a parameter or injected field. Hard-coding the choice is DIP=0.
(b) Static singleton access to replaceable behaviour.
MailSender.getInstance().send(...), DatabaseRegistry.getInstance().getConnection(). The static-singleton pattern locks the method to one implementation. This is DIP=0.
(c) Tightly coupled framework implementation choices.
new MyAppSpecificRepository() when the method ought to depend on a Repository interface. The method is hard-coding a low-level choice the framework could otherwise abstract.

5. Worked examples — the canonical patterns

Example A — DIP = 2 (Compliant)

public BigDecimal totalPrice(List<LineItem> items) {
    BigDecimal total = BigDecimal.ZERO;
    for (LineItem item : items) {
        total = total.add(item.price());
    }
    return total;
}
Score: 2

No replaceable dependencies. The method takes a list as a parameter (dependency arrives from outside), uses BigDecimal.ZERO (value type), and calls a method on the parameter. There is no new at all, and even if BigDecimal.ZERO were new BigDecimal(0), that is a value type — not a DIP concern.

Example B — DIP = 0 (Violated)

public void notifyCustomer(String email, String message) {
    SmtpMailSender sender = new SmtpMailSender("mail.example.com", 587);
    sender.send(email, message);
}
Score: 0

The method directly instantiates a specific mail-sending service. SmtpMailSender is a replaceable collaborator — the method could equally well use a SendgridMailSender or a test fake. By instantiating it directly, the method hard-codes the low-level choice. This is the canonical DIP violation Martin describes.

Example C — DIP = 2 (Compliant)

public void notifyCustomer(MailSender sender, String email, String message) {
    sender.send(email, message);
}
Score: 2

Same intent as Example B, but the mailer arrives as a parameter. The method depends on an abstraction (or at least on something injected from outside). The caller chooses the implementation. No DIP violation.

Example D — DIP = 0 (Violated, static-singleton form)

public void recordOrder(Order order) {
    DatabaseRegistry.getInstance().save(order);
}
Score: 0

The method reaches into a global static singleton for a replaceable collaborator (the database). There is no way for a caller to substitute a different database, an in-memory test version, or a mock. This is DIP=0 even though no new appears in the method body.

Example E — DIP = 2 (Compliant, despite multiple new)

public List<String> readLines(Path file) throws IOException {
    List<String> result = new ArrayList<>();
    for (String line : Files.readAllLines(file)) {
        result.add(line.trim());
    }
    return result;
}
Score: 2

new ArrayList<>() is a collection, not a replaceable collaborator. Files.readAllLines is a JDK static utility, not a replaceable service. The method has zero DIP violations even though it allocates and calls a static API.

Example F — DIP = 1 (Uncertain)

public Report generate(ReportRequest request) {
    Formatter formatter = new Formatter();
    return formatter.format(request);
}
Score: 1

The method instantiates a Formatter. Without seeing the class definition, we cannot tell whether Formatter is a replaceable service (in which case DIP=0) or a thin local helper (in which case DIP=2). The name is generic, and the constructor takes no arguments. When the method alone does not give us enough signal, the rubric tells us to score 1.

Example G — DIP = 2 (Compliant constructor)

public Customer(String name, String email) {
    this.name = Objects.requireNonNull(name);
    this.email = Objects.requireNonNull(email);
    this.createdAt = Instant.now();
}
Score: 2

A constructor body. The method's job is to construct an object. Objects.requireNonNull is a JDK utility, Instant.now() is a static factory for a value type. No replaceable collaborators are hard-coded.

Example H — DIP = 0 (Violated, exception-disguised)

public Connection openConnection() {
    return new MySqlConnection("jdbc:mysql://prod-db:3306/orders");
}
Score: 0

The method instantiates a specific connection implementation with a hard-coded URL. The connection type is replaceable (could be PostgresConnection, H2Connection for tests, etc.). Both the choice of implementation AND the choice of database URL are hard-coded inside the method.

6. Common boundary mistakes to avoid

(a) Treating "any new" as DIP violation.
The new keyword alone does not violate DIP. The question is what is being instantiated. Collections, value types, exceptions, and helpers are fine. Replaceable services are not.
(b) Marking 0 when the type's role is unclear.
If the method instantiates SomeHelper and you cannot tell from the method alone whether SomeHelper is a replaceable service or a value object, the rubric says to score 1, not 0. Save score 0 for cases where the violation is clear from the method's text.
(c) Treating chained method calls on parameters as DIP violations.
logicalConnection.afterStatement() where logicalConnection is a parameter is not a DIP violation — the dependency arrived from outside. The method is depending on whatever the caller passed in.
(d) Treating exceptions as DIP violations.
throw new IllegalArgumentException(...) is part of the method's contract, not a replaceable collaborator. Exception construction never counts as DIP=0.
(e) Treating static JDK calls as DIP violations.
Files.copy(...), Arrays.sort(...), Objects.requireNonNull(...), Math.min(...), String.format(...) are namespaced functions, not services. They do not count.

7. Decision rule (quick reference)

When scoring a method on DIP, ask in order:

  1. Does the method create or call any object that has plausible alternative implementations another application would want to plug in?
  2. If yes, is that object instantiated inside the method with new, accessed via a static singleton, or arrived as a parameter / field?
  3. Instantiated inside or static singleton → score 0. Arrived from outside → score 2.
  4. If the type's role is unclear from the method alone → score 1.
  5. If the method has no objects matching the description in (1) at all → score 2.

8. Re-annotation instructions

Important:

  • Re-score only the DIP column for each of the 200 methods. SRP and OCP scores from round 2 are final.
  • Score independently. Do not look at the other rater's scores. Do not look at your own round-1 or round-2 scores either.
  • Apply the decision rule in §7 to each method.
  • When in doubt, score 1. Score 1 is the right answer when the method alone does not give you enough signal.
  • Use the comment field for unusual cases.
  • Allow 1–2 minutes per method. Total time should be around 3–4 hours.
